# 自媒体运营对话入口规范

## 新主线

从现在开始，这套工具的主入口改为：

1. 用户通过对话发起操作
2. 系统返回对应动作、登录入口或下一步指引
3. 用户完成登录后回复一句“已完成”
4. 系统继续执行抓数或后续动作
5. 数据结果按用户选择，统一沉淀到电脑本地文件或飞书展示

## 安装后的首次引导

当这个能力被封装成 skill 后，用户安装完成时，Codex 必须先完成一次初始化对话。

### 1. 安装完成后必须回复

用几句话告诉用户：

1. 这个 skill 用来统一管理小红书、抖音、微信视频号账号。
2. 它可以抓账号数据、笔记/作品数据、直播数据。
3. 新增账号可以说：新增小红书 / 新增抖音 / 新增微信视频号。
4. 手动拉数可以说：抓取所有平台数据。
5. 默认定时任务是每天24点（00:00）自动刷新。
6. 修改定时任务可以说：定时改成 09:00，然后根据提示选择范围。

最后必须询问用户：

> 你的数据想存在哪里？请回复：电脑 或 飞书。

### 2. 用户选择“电脑”

Codex 必须执行：

1. 创建本地数据文件夹，默认名称为“自媒体运营数据”。
2. 在文件夹里创建“账号数据.csv”和“内容直播数据.csv”。
3. 后续新增账号、首次拉数、手动拉数、定时拉数，都更新这两个文件。
4. 回复用户这两个文件的完整路径。
5. 说明每个文件存储什么：
   - 账号数据.csv：平台、账号名称、账号ID、粉丝数、获赞数、笔记/作品数、最近拉数时间、数据状态等。
   - 内容直播数据.csv：笔记/作品数据和直播数据，用“分类”字段区分内容或直播。
6. 最后引导用户新增账号。

推荐回复口径：

> 已设置为存到这台电脑，并创建好数据文件夹。
> 账号数据会更新到：xxx
> 内容直播数据会更新到：xxx
> 下一步可以直接说：新增小红书 / 新增抖音 / 新增微信视频号。

### 3. 用户选择“飞书”

Codex 必须执行：

1. 检查或安装飞书 CLI。
2. 输出飞书授权页面，让用户完成授权。
3. 明确告诉用户：这个授权只用于后续把账号数据、内容数据和直播数据同步到飞书。
4. 用户回复“飞书授权完成”后，创建知识库“自媒体运营”。
5. 在知识库中创建“账号数据”和“内容／直播数据”两个表格。
6. 后续新增账号、首次拉数、手动拉数、定时拉数，都同步到飞书。
7. 回复用户知识库名称、表格名称、每个表格存储什么。
8. 最后引导用户新增账号。

推荐回复口径：

> 已设置为同步到飞书。
> 知识库：自媒体运营
> 账号数据：记录平台账号、账号数据和状态。
> 内容／直播数据：记录笔记/作品数据和直播数据。
> 下一步可以直接说：新增小红书 / 新增抖音 / 新增微信视频号。

## 用户常见指令

### 新增账号类

- 增加小红书账号
- 增加抖音账号
- 增加微信视频号账号
- 给我新增一个小红书账号
- 我要接入一个新的抖音号

### 登录确认类

- 登录完成了
- 小红书登录好了
- 抖音第二个账号登录好了
- 视频号已经登录完成

### 抓数类

- 抓一下今天的数据
- 手动抓一下小红书
- 抓全部平台
- 抓某个账号的数据

### 异常类

- 这个账号异常了
- 重新登录这个号
- 重新抓一下
- 看看哪个账号有问题

## 系统返回结构

### 1. 新增账号时

系统需要返回：

1. 已识别的平台
2. 建议使用的账号标识
3. 本次登录入口
4. 登录完成后让用户回复的话

示例口径：

> 已为你准备好小红书账号登录入口。  
> 请在弹出的浏览器里完成登录。  
> 登录完成后，直接回复我：小红书账号已登录完成。

### 2. 登录完成确认后

系统需要执行：

1. 校验本地登录态是否存在
2. 如存在，触发该账号真实抓数
3. 抓数成功后回写飞书展示层

### 3. 抓数完成后

系统需要返回：

1. 哪个平台
2. 哪个账号
3. 是否成功
4. 抓到哪些数据类型
5. 飞书里去哪里查看

## 展示/存储层要求

电脑本地或飞书里最终要统一展示/存储：

1. 已登录的平台和账号
2. 登录状态
3. 最近一次抓取时间
4. 账号数据
5. 内容数据
6. 直播数据
7. 异常状态

## 本地执行器要求

本地执行器负责：

1. 打开浏览器登录
2. 保存登录态
3. 触发单账号抓数
4. 触发多账号抓数
5. 跑定时任务

## 当前优先实现顺序

1. 新增账号 -> 返回登录入口
2. 用户回复“登录完成” -> 触发真实抓数
3. 抓数结果统一落盘
4. 飞书展示账号状态和数据结果
