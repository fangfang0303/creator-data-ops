自媒体运营工具总目标

一、产品定位
做一套给中小商家、自媒体创作者、小团队使用的轻量运营工具。
目标不是做复杂的大系统，而是做一套用户容易上手、能省人工、后续可以持续交付和收费的产品。

二、核心平台范围
当前核心平台包括：
1. 小红书
2. 抖音
3. 微信视频号

三、核心能力范围
围绕每个平台、每一个账号，持续建设以下能力：
1. 多平台、多账号统一管理
2. 账号登录状态管理
3. 账号数据获取
4. 笔记/内容数据获取
5. 直播数据获取
6. 定时自动拉数
7. 手动触发拉数
8. 异常处理，包括重新登录、重新抓取、状态回写
9. 统一上传内容到指定平台和指定账号

四、最新主线结构
从现在开始，整体产品主线不再以“飞书页面直接联动本地动作”为优先方向，
而是改为以下三层结构：

1. 对话入口层
- 用户通过对话发起操作
- 例如：增加小红书账号、增加抖音账号、抓取今天数据、重新登录某个账号
- 系统根据用户指令返回下一步动作
- 如果需要登录，则直接返回登录入口或登录指引

2. 数据展示/存储层
- 用户可选择数据存储在电脑本地，或同步到飞书
- 飞书不再强承担所有本地执行动作
- 电脑本地和飞书都主要负责统一展示结果
- 展示/存储内容包括：
  - 已登录了哪些平台、哪些账号
  - 哪些账号异常
  - 账号数据
  - 笔记/内容数据
  - 直播数据
  - 最近一次抓取时间

3. 本地执行器层
- 用户电脑本地必须保留一个轻执行器
- 本地执行器负责：
  - 打开浏览器登录
  - 保存登录态
  - 抓取数据
  - 跑定时任务
  - 后续执行统一发布

五、最终使用方式
最终目标不是让用户频繁打开很多脚本和页面。
最终形态应当是：
1. 用户平时主要通过对话发起操作
2. 我根据指令返回对应动作和登录入口
3. 用户完成登录后回复一句“已登录完成”或类似确认
4. 我继续触发后续抓数
5. 所有结果按用户选择，统一沉淀到电脑本地文件或飞书里展示

六、产品形态方向
当前优先做本地执行版，把能力跑通、跑稳、可交付。
后续再根据业务发展，演进为更轻交付、门槛更低的版本，例如：
1. 安装包版本
2. 半托管版本
3. SaaS 版本

七、做事原则
后续所有功能开发、流程设计、页面调整，都优先围绕这个总目标判断：
1. 是否有助于多平台、多账号统一运营
2. 是否有助于“对话作为操作入口”
3. 是否有助于“电脑本地或飞书作为统一展示/存储层”
4. 是否有助于降低用户使用门槛
5. 是否有助于提升稳定性
6. 是否有助于后续商业化交付

八、当前阶段主线任务
当前阶段优先级如下：
1. 把小红书、抖音、微信视频号的多账号登录和抓数链路做稳定
2. 把账号数据、笔记数据、直播数据尽可能打通
3. 把“用户发一句话 -> 返回登录入口/执行动作 -> 用户确认 -> 继续抓数”这套对话主链做通
4. 把数据展示/存储层做稳定，支持电脑本地文件或飞书展示账号状态、数据结果、异常状态
5. 把定时任务、手动拉数、异常处理做完整
6. 把统一上传内容能力接入

九、后续沟通约定
后续所有沟通，默认以这份总目标为准。
如遇到局部细节与总目标冲突，优先服从总目标。

十、定时任务对话规则
当用户通过对话要求修改定时任务时间时，不直接默认修改全部账号。
必须先确认本次修改范围：
1. 所有平台的所有账号
2. 某个平台的所有账号
3. 某个平台下的某一个账号
只有时间和范围都确认后，才更新对应的本地计划任务。

十一、飞书展示层最新规则
飞书不再承担登录、授权、拉数触发入口，不再做登录链接和监听。
飞书只负责展示本地执行器已经抓到的数据。
标准结构如下：
1. 初始创建知识库：自媒体运营
2. 知识库中只保留两张核心多维表格：账号数据、内容／直播数据
3. 新增平台账号后，自动更新账号数据表
4. 新增平台账号并完成首次拉数后，自动更新内容／直播数据表
5. 手动拉数、定时拉数完成后，也自动同步更新内容／直播数据表
6. 账号数据表字段以当前能获取到的账号维度字段为准
7. 内容／直播数据表字段以当前能获取到的内容和直播字段为准，并增加分类字段，分类值为内容或直播
8. 统一发布内容能力先暂停，不作为当前阶段主线

十二、未来 Skill 安装后的初始化规则
后期这套工具会封装成一个可复用 skill，给需要的用户安装使用。
安装后的标准流程如下：
1. 用户说“安装 xx skill”，Codex 负责完成 skill 安装。
2. 安装完成后，Codex 必须先用几句话说明这个 skill 是做什么的：
   - 统一管理小红书、抖音、微信视频号账号
   - 自动抓账号数据、笔记/作品数据、直播数据
   - 支持新增平台账号、手动拉数、定时自动拉数、修改定时任务时间
3. Codex 必须告诉用户：
   - 新增账号可以说：新增小红书 / 新增抖音 / 新增微信视频号
   - 手动拉数可以说：抓取所有平台数据
   - 默认定时任务时间是每天24点（00:00）
   - 修改定时任务可以说：定时改成 09:00，然后继续选择范围
4. 安装说明后，Codex 必须询问用户：数据是存储在电脑上，还是飞书上。
5. 如果用户回复“电脑”：
   - Codex 创建一个本地数据文件夹
   - 在文件夹中创建“账号数据”和“内容直播数据”两个表格文件
   - 后续新增账号、首次拉数、手动拉数、定时拉数都更新这些本地文件
   - Codex 回复每个文件的路径，以及每个文件存储什么数据
   - 最后引导用户新增平台账号
6. 如果用户回复“飞书”：
   - Codex 先安装/检查飞书 CLI 能力
   - 输出飞书授权页面，让用户完成授权
   - 明确告知授权仅用于后续同步飞书数据
   - 用户回复“授权完成”后，Codex 创建知识库“自媒体运营”
   - 在知识库中创建“账号数据”和“内容／直播数据”两个表格
   - 后续新增账号、首次拉数、手动拉数、定时拉数都同步到飞书
   - Codex 回复知识库名称、表格名称、每个表格存储什么数据
   - 最后引导用户新增平台账号
